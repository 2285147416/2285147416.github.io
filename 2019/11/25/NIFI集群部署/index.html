#概念：

协调节点：coordinator node, 所有节点往该节点发送心跳信息
主节点：primary node, 主要用于跑一些独立流程
独立流程：isolated processors, 有些流程需要独立跑，不要在集群中运行，例如getsftp流程，该流程需要从FTP服务器上获取文件信息，如果多个流程同时运行则会影响文件的获取一致性。

Zookeeper服务集群配置
Nifi集群的协调配置管理需要第三方协调配置服务，即zookeeper集群。 可以使用环境中已经搭建好的zookeeper服务，NIFI本身也集成了zookeeper服务，我们采用NIFI自身启动的Zookeeper。三个NIFI节点集群，同时也相应搭建三个节点的Zookeeper服务集群。为此需要修改一下配置：
修改$NIFI_HOME/conf/nifi.properties
nifi.state.management.embedded.zookeeper.start=true //启用nifi中的zookeeper服务
nifi.state.management.embedded.zookeeper.properties=./conf/zookeeper.properties //zookeeperf服务集群配置信息，即集群的配置在zookeeper.properties中定义

#集群状态管理
我们知道nifi中所有数据流的处理会涉及到状态保存，例如从FTP取文件，会保存文件的最新时间，以便于下次取数据的时候取最新时间下的文件信息。默认单点情况下数据流的状态都保存在服务节点本地，但是在启用集群后，需要将这些状态信息保存在第三方公共服务区，这样以便集群中所有服务节点可以获取到并且同步状态信息。而这个第三方服务一般是采用zookeeper进行管理。为此需要实现以下几点配置：
nifi.state.management.configuration.file=./conf/state-management.xml 状态管理配置文件 采用默认即可
nifi.state.management.provide.cluster=zk-provider 状态管理采用zk-provider方式

#修改$NIFI_HOME/conf/state-management.xml文件，找到并且id为zk-provider的标签。

修改Connect String 为 dn1:2181,dn2:2181,dn3:2181
修改Root Node 为 /nifi

#集群节点属性配置
修$NIFI_HOME/conf/nifi.properties中以下配置：

nifi.cluster.is.node=true
nifi.cluster.node.address=dn1
nifi.zookeeper.connect.string=dn1:2181,dn2:2181,dn3:2181
nifi.zookeeper.root.node=/nifi
nifi.remote.input.host=dn3
nifi.remote.input.socket.port=9998
Nifi.web.http.host=dn3

STEP SIX 启动各个节点
在各个服务节点上，进入$NIFI_HOME目录，执行bin/nifi.sh start进行各个节点启动。
访问任意节点(以dn1节点为例): http://dn1:8080/nifi
